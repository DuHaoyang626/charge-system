# 数据设计说明

> **文档状态**：定稿  
> **对应阶段**：编码前数据设计  
> **技术栈**：SQLite (SQLAlchemy ORM) + application.yml (PyYAML)  
> **更新日期**：2026-06-07

---

## 一、概述

本系统数据存储采用 **配置文件（静态参数）+ 数据库（动态持久化数据）** 分层策略：

| 存储位置 | 数据类型 | 特点 |
|----------|----------|------|
| `application.yml` | 充电桩硬件参数、容量配置、默认调度策略、默认计费规则、电价时段定义、日志阈值 | 系统启动时一次性读取，运行中可通过管理员API在线修改（修改后同时更新内存和配置文件） |
| SQLite 数据库 | 用户账号、车辆信息、充电桩动态数据、充电请求、充电会话、账单记录、故障事件、系统日志 | 持续增长的业务数据，通过 SQLAlchemy ORM 访问 |

### 1.1 设计原则

1. **配置与数据分离**：静态的、环境相关的参数放配置文件；动态的、与业务实体关联的数据放数据库
2. **配置可运行时修改**：管理员通过 `setParameters` 修改的参数（如电价、计费规则）会同步更新配置文件和内存缓存，重启不丢失
3. **数据库只存业务数据**：不存硬件拓扑、默认策略等配置类信息
4. **每桩独立计费**：电价和服务费可在充电桩级别独立配置（`pile_tariff_configs` 表），若某桩无独立配置则使用配置文件中的全局默认值

## 二、配置文件设计（application.yml）

### 2.1 文件位置与加载机制

- **文件路径**：项目根目录 `config/application.yml`
- **加载时机**：系统启动时由 `ConfigLoader` 读取，初始化 `DispatchStrategy`、`ChargingStation` 等模块
- **修改机制**：
  - 管理员通过 `PUT /api/piles/{pile_Id}/parameters` 修改电价和计费规则时，同步写回配置文件
  - 管理员通过 `PUT /api/strategies/dispatch`、`PUT /api/strategies/fault` 切换策略时，同步写回配置文件
  - 修改后调用 `ConfigLoader.reload()` 刷新内存缓存

### 2.3 完整配置模板

```yaml
# ============================================
# 智能充电桩调度计费系统 — 配置文件
# 系统启动时自动读取，运行时可通过 API 修改
# ============================================

station:
  name: "智能充电站-01"
  max_queue_capacity: 50       # 每个充电桩排队区最大车辆数（超出拒绝入队）
  max_waiting_capacity: 5      # 每个充电桩等待区最大车辆数

# ------------------------------------------------------------------
# 充电桩配置
# 每个充电桩独立配置：类型、功率、支持协议、充电位数量
# 新增充电桩只需在此处添加条目，重启后自动识别
# ------------------------------------------------------------------
piles:
  - id: "P001"
    name: "快充桩-01"
    type: fast_charge           # fast_charge | slow_charge
    max_power_kw: 120.0         # 最大输出功率（kW）
    protocols:                  # 支持的充电协议列表
      - GB_STANDARD
      - CCS
    parking_spots: 2            # 充电区占用的物理停车位数量（默认2个）

  - id: "P002"
    name: "快充桩-02"
    type: fast_charge
    max_power_kw: 120.0
    protocols:
      - GB_STANDARD
      - CCS
      - CHADEMO
    parking_spots: 2

  - id: "P003"
    name: "慢充桩-01"
    type: slow_charge
    max_power_kw: 7.0
    protocols:
      - GB_STANDARD
    parking_spots: 1

  - id: "P004"
    name: "慢充桩-02"
    type: slow_charge
    max_power_kw: 7.0
    protocols:
      - GB_STANDARD
      - CCS
    parking_spots: 1

  - id: "P005"
    name: "慢充桩-03"
    type: slow_charge
    max_power_kw: 11.0
    protocols:
      - GB_STANDARD
      - CHADEMO
    parking_spots: 1

# ------------------------------------------------------------------
# 调度策略配置
# 系统启动时加载此处的默认值
# 管理员运行中可通过 API 切换，切换时同步写回此处
# ------------------------------------------------------------------
dispatch:
  # 正常分配策略：SHORTEST_TOTAL_TIME | BATCH_SHORTEST_TIME
  default_algorithm: "SHORTEST_TOTAL_TIME"

  # 故障处理策略：PRIORITY | TIME_ORDER | FAULT_RECOVERY
  #              | SHORTEST_TOTAL_TIME | BATCH_SHORTEST_TIME
  default_fault_strategy: "TIME_ORDER"

# ------------------------------------------------------------------
# 计费规则配置（全局默认值）
# 充电桩级别独立配置存储在数据库 pile_tariff_configs 表中
# 若某桩无独立配置，则使用此处全局默认值
# ------------------------------------------------------------------
billing:
  # --- 时段定义（24小时制，闭区间 [start, end)）---
  time_periods:
    peak:
      - start: "10:00"
        end: "12:00"
      - start: "17:00"
        end: "19:00"
    valley:
      - start: "00:00"
        end: "08:00"
      - start: "22:00"
        end: "23:59"
    normal:                    # 平时为峰时和谷时之外的剩余时段
      - start: "08:00"
        end: "10:00"
      - start: "12:00"
        end: "17:00"
      - start: "19:00"
        end: "22:00"

  # --- 默认三时段电价（元/kWh）---
  default_prices:
    peak_price: 1.2            # 峰时电价
    normal_price: 0.8          # 平时电价
    valley_price: 0.4          # 谷时电价

  # --- 服务费规则 ---
  service_fee:
    base_fee: 5.0              # 基础服务费（元），等待区退出/充电区超时扣除
    time_coefficient: 0.5      # 时长系数（元/分钟）
    overtime_penalty: 0.8      # 超时充电附加费率（元/分钟）

  # --- 超时配置 ---
  timeout_config:
    charging_confirm_timeout_seconds: 120   # 进入充电区确认超时（秒）
    timeout_base_fee: 5.0                   # 超时取消扣除的基本服务费（元，可每桩不同，此处为默认值）

# ------------------------------------------------------------------
# 监控配置
# ------------------------------------------------------------------
monitoring:
  refresh_interval_seconds: 10  # 管理员客户端轮询刷新间隔（秒）

# ------------------------------------------------------------------
# 日志配置
# 日志等级：0=常规, 1=注意, 2=警告, 3=严重, 4=崩溃
# ------------------------------------------------------------------
logging:
  # 终端输出阈值：记录到数据库的日志中，高于此等级的额外输出到终端
  # 例如设为 1（注意），则等级 ≥1 的日志既入 DB 也输出到终端；
  # 设为 4（崩溃），则仅等级 4 输出到终端；
  # 设为 -1，则所有日志仅入 DB，不输出终端
  console_threshold: 1

# ------------------------------------------------------------------
# 系统通用配置
# ------------------------------------------------------------------
system:
  jwt_secret: "charge-system-jwt-secret-key-2026"
  token_expire_minutes: 1440    # Token 过期时间（24小时）
  db_path: "data/charge_system.db"  # 数据库文件路径（相对于项目根目录）
```

## 三、数据库表设计

### 3.1 表清单

| 序号 | 表名 | 中文名 | 对应实体 | 说明 |
|------|------|--------|----------|------|
| 1 | `users` | 用户表 | User | 用户账号信息（含登录凭据） |
| 2 | `vehicles` | 车辆表 | Vehicle | 车辆信息，与用户一对多 |
| 3 | `charging_piles` | 充电桩表 | ChargingPile | 充电桩运行时状态与累计数据 |
| 4 | `pile_protocols` | 充电桩协议表 | — | 充电桩与协议的关联（多对多） |
| 5 | `pile_tariff_configs` | 充电桩计费规则表 | TariffConfig | 每桩独立的计费规则配置 |
| 6 | `charging_requests` | 充电请求表 | ChargingRequest | 充电请求 + 三区队列状态 |
| 7 | `charging_sessions` | 充电会话表 | ChargingSession | 一次充电的完整记录 |
| 8 | `billing_records` | 账单记录表 | BillingRecord | 充电费用概览 |
| 9 | `detailed_bills` | 详单记录表 | DetailedBill | 分时段费用明细 |
| 10 | `payment_orders` | 支付订单表 | PaymentOrder | 支付状态记录 |
| 11 | `fault_events` | 故障事件表 | FaultEvent | 充电桩故障事件记录 |
| 12 | `system_logs` | 系统日志表 | — | 统一系统日志（全模块、全等级） |
| 13 | `dispatch_strategy_logs` | 策略变更日志表 | — | 调度策略切换记录 |

> **ER 关系概要**：
> - `users` 1→N `vehicles`：一个用户可有多辆车
> - `vehicles` 1→N `charging_requests`：一辆车可发起多次充电请求
> - `charging_piles` 1→N `charging_requests`：一个充电桩的队列中可有多条请求
> - `charging_piles` M→N `pile_protocols`：充电桩与协议多对多
> - `charging_piles` 1→1 `pile_tariff_configs`：每桩至多一条独立计费规则（可选）
> - `charging_requests` 1→1 `charging_sessions`：一条请求对应一次（或零次）会话
> - `charging_sessions` 1→1 `billing_records`：一次会话生成一条账单
> - `billing_records` 1→N `detailed_bills`：一条账单有多条分段详单
> - `billing_records` 1→1 `payment_orders`：一条账单对应一个支付订单
> - `charging_piles` 1→N `fault_events`：一个充电桩可有多条故障记录

### 3.2 表结构详述

---

#### 表 1：`users` — 用户账号表

存储用户登录账号信息和基本资料。

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PK, AUTO INCREMENT | 内部自增主键 |
| `user_id` | VARCHAR(32) | UNIQUE, NOT NULL | 用户编号（业务标识，如 "U20260001"） |
| `user_name` | VARCHAR(50) | NOT NULL | 用户名（注册时填写） |
| `license_plate` | VARCHAR(20) | UNIQUE, NOT NULL | 车牌号，也作为登录账号 |
| `password_hash` | VARCHAR(128) | NOT NULL | 密码哈希值（bcrypt 加密） |
| `membership_level` | INTEGER | NOT NULL, DEFAULT 0 | 会员等级（0=普通，1=银卡，2=金卡，3=钻石） |
| `account_status` | VARCHAR(16) | NOT NULL, DEFAULT 'INACTIVE' | 账号状态：`INACTIVE`（未激活/未设置密码）, `ACTIVE`（已激活）, `LOCKED`（已锁定） |
| `created_at` | DATETIME | NOT NULL | 注册时间 |
| `updated_at` | DATETIME | NOT NULL | 最后更新时间 |
| `last_login_at` | DATETIME | NULL | 最后登录时间 |

**索引**：`idx_users_license_plate` 唯一索引 on `license_plate`

**示例数据**：
```json
{
  "id": 1,
  "user_id": "U20260001",
  "user_name": "张三",
  "license_plate": "京A12345",
  "password_hash": "$2b$12$...",
  "membership_level": 1,
  "account_status": "ACTIVE",
  "created_at": "2026-06-01 10:30:00",
  "updated_at": "2026-06-07 14:20:00",
  "last_login_at": "2026-06-07 09:15:00"
}
```

---

#### 表 2：`vehicles` — 车辆信息表

记录每辆车的技术参数。

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PK, AUTO INCREMENT | 内部自增主键 |
| `vehicle_id` | VARCHAR(32) | UNIQUE, NOT NULL | 车辆编号（业务标识，如 "V20260001"） |
| `user_id` | VARCHAR(32) | NOT NULL, FK → users(user_id) | 所属用户编号 |
| `license_plate` | VARCHAR(20) | UNIQUE, NOT NULL | 车牌号（与 users.license_plate 一致，冗余便于查询） |
| `battery_capacity_kwh` | DECIMAL(8,2) | NOT NULL | 电池总容量（kWh） |
| `current_battery_percentage` | DECIMAL(5,2) | NOT NULL, DEFAULT 0 | 当前电量百分比（0.00~100.00） |
| `charging_protocol` | VARCHAR(20) | NOT NULL | 车辆支持的充电协议：`GB_STANDARD`, `CCS`, `CHADEMO` |
| `created_at` | DATETIME | NOT NULL | 创建时间 |
| `updated_at` | DATETIME | NOT NULL | 最后更新时间 |

**索引**：`idx_vehicles_user_id` on `user_id`  
**检查约束**：`current_battery_percentage BETWEEN 0 AND 100`

---

#### 表 3：`charging_piles` — 充电桩表

存储充电桩的运行状态和累计统计数据。充电桩的硬件参数（功率、协议、类型等）存储在 `application.yml` 中，此表仅记录动态运行数据。

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PK, AUTO INCREMENT | 内部自增主键 |
| `pile_id` | VARCHAR(16) | UNIQUE, NOT NULL | 充电桩编号（如 "P001"），与配置文件中的 id 对应 |
| `pile_name` | VARCHAR(50) | NOT NULL | 充电桩名称，与配置文件同步 |
| `type` | VARCHAR(16) | NOT NULL | 类型：`fast_charge` / `slow_charge` |
| `max_power_kw` | DECIMAL(6,2) | NOT NULL | 最大功率（kW） |
| `status` | VARCHAR(16) | NOT NULL, DEFAULT 'AVAILABLE' | 运行状态：`AVAILABLE`（可用/空闲）, `QUEUEING`（排队中）, `CHARGING`（充电中）, `RUNNING`（运行中）, `CLOSED`（关闭）, `FAULT`（故障） |
| `total_charge_num` | INTEGER | NOT NULL, DEFAULT 0 | 累计充电次数 |
| `total_charge_time` | DECIMAL(10,2) | NOT NULL, DEFAULT 0 | 累计充电时长（分钟） |
| `total_capacity` | DECIMAL(12,2) | NOT NULL, DEFAULT 0 | 累计充电容量（kWh） |
| `parking_spots` | INTEGER | NOT NULL, DEFAULT 1 | 充电区占用停车位数量 |
| `current_charging_count` | INTEGER | NOT NULL, DEFAULT 0 | 当前正在充电的车辆数 |
| `created_at` | DATETIME | NOT NULL | 入库时间 |
| `updated_at` | DATETIME | NOT NULL | 最后更新时间 |

**索引**：`idx_piles_status` on `status`

> **说明**：`pile_id` 与配置文件中的 id 保持一致。系统启动时扫描配置文件中的 pile 列表，将新增的桩插入此表，已存在的桩用配置文件中的硬件参数覆盖更新 `max_power_kw`、`type` 等字段。

---

#### 表 4：`pile_protocols` — 充电桩支持协议关联表

充电桩与充电协议的多对多关系。每个充电桩可支持多种协议。

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PK, AUTO INCREMENT | 内部自增主键 |
| `pile_id` | VARCHAR(16) | NOT NULL, FK → charging_piles(pile_id) | 充电桩编号 |
| `protocol` | VARCHAR(20) | NOT NULL | 协议类型：`GB_STANDARD`, `CCS`, `CHADEMO` |
| `created_at` | DATETIME | NOT NULL | 创建时间 |

**索引**：`UNIQUE(pile_id, protocol)` 联合唯一索引

---

#### 表 5：`pile_tariff_configs` — 充电桩计费规则表

存储每个充电桩独立的计费规则。若某桩无记录，则使用配置文件 `billing` 中的全局默认值。

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PK, AUTO INCREMENT | 内部自增主键 |
| `pile_id` | VARCHAR(16) | UNIQUE, NOT NULL, FK → charging_piles(pile_id) | 充电桩编号 |
| `peak_price` | DECIMAL(6,4) | NOT NULL, DEFAULT 1.2 | 峰时电价（元/kWh） |
| `normal_price` | DECIMAL(6,4) | NOT NULL, DEFAULT 0.8 | 平时电价（元/kWh） |
| `valley_price` | DECIMAL(6,4) | NOT NULL, DEFAULT 0.4 | 谷时电价（元/kWh） |
| `base_service_fee` | DECIMAL(6,2) | NOT NULL, DEFAULT 5.0 | 基础服务费（元） |
| `time_coefficient` | DECIMAL(6,4) | NOT NULL, DEFAULT 0.5 | 时长系数（元/分钟） |
| `overtime_penalty` | DECIMAL(6,4) | NOT NULL, DEFAULT 0.8 | 超时罚金（元/分钟） |
| `timeout_base_fee` | DECIMAL(6,2) | NOT NULL, DEFAULT 5.0 | 超时取消扣除的基本服务费（元） |
| `created_at` | DATETIME | NOT NULL | 创建时间 |
| `updated_at` | DATETIME | NOT NULL | 最后更新时间 |

**索引**：`idx_tariff_pile_id` on `pile_id`（唯一）

---

#### 表 6：`charging_requests` — 充电请求表

用户每次提交充电请求生成一条记录，贯穿排队→等待→充电全生命周期。此表同时承担三区队列的管理职能。

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PK, AUTO INCREMENT | 内部自增主键 |
| `request_id` | VARCHAR(32) | UNIQUE, NOT NULL | 请求编号（业务标识，如 "R20260607001"） |
| `car_id` | VARCHAR(20) | NOT NULL, FK → vehicles(license_plate) | 车牌号（对应车辆） |
| `pile_id` | VARCHAR(16) | NOT NULL, FK → charging_piles(pile_id) | 当前所属充电桩编号 |
| `request_time` | DATETIME | NOT NULL | 请求提交时间 |
| `charging_mode` | VARCHAR(16) | NOT NULL | 充电模式：`FAST_CHARGE` / `SLOW_CHARGE` |
| `target_power_kwh` | DECIMAL(8,2) | NOT NULL | 目标充电电量（kWh） |
| `request_status` | VARCHAR(16) | NOT NULL, DEFAULT 'QUEUED' | 请求状态：`QUEUED`（排队中）, `WAITING`（等待区中）, `CHARGING`（充电中）, `COMPLETED`（已完成）, `CANCELLED`（已取消） |
| `zone_type` | VARCHAR(16) | NOT NULL, DEFAULT 'QUEUE_AREA' | 当前所在区域：`QUEUE_AREA`（排队区）, `WAITING_AREA`（等待区）, `CHARGING_AREA`（充电区） |
| `queue_position` | INTEGER | NULL | 在当前队列中的位置序号（从1开始，NULL表示非严格排序场景） |
| `original_pile_id` | VARCHAR(16) | NULL | 最初分配的充电桩编号（换队后记录原始桩，用于跟踪） |
| `is_active` | BOOLEAN | NOT NULL, DEFAULT TRUE | 是否活跃（逻辑删除标记，排队中/等待中/充电中为TRUE；完成/取消为FALSE） |
| `created_at` | DATETIME | NOT NULL | 创建时间 |
| `updated_at` | DATETIME | NOT NULL | 最后更新时间 |

**索引**：
- `idx_requests_car_id` on `car_id`
- `idx_requests_pile_id_status` on `(pile_id, request_status)`
- `idx_requests_car_id_status` on `(car_id, request_status)` — 查询用户当前活跃请求

> **说明**：
> - 一个车辆同一时间只能有一条 `request_status IN ('QUEUED','WAITING','CHARGING')` 的活跃请求
> - `zone_type` 反映车辆在三区中的当前位置：`QUEUE_AREA` → `WAITING_AREA` → `CHARGING_AREA`（单向流转）
> - 换队操作（排队区内）：仅修改 `pile_id`，排至目标队列队尾，`request_status` 不变

#### 表 12：`system_logs` — 系统日志表

统一记录全系统所有模块产生的日志，五级分类，全部持久化到数据库。根据配置文件 `logging.console_threshold` 的设置，高于该阈值的日志同时输出到终端。

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INTEGER | PK, AUTO INCREMENT | 内部自增主键 |
| `log_id` | VARCHAR(32) | UNIQUE, NOT NULL | 日志编号（如 "LOG20260607001"） |
| `module` | VARCHAR(32) | NOT NULL | 发送日志的模块名，如 `SYSTEM`, `ACCOUNT`, `DISPATCH`, `QUEUE`, `BILLING`, `MONITOR`, `ADMIN`, `FAULT`, `SCHEDULER`, `API` |
| `level` | INTEGER | NOT NULL | 日志等级：`0`=常规, `1`=注意, `2`=警告, `3`=严重, `4`=崩溃 |
| `detail` | TEXT | NOT NULL | 日志详细信息，建议格式：`"[操作] 描述 (关键参数: 值)"` |
| `created_at` | DATETIME | NOT NULL | 日志生成时间 |

**索引**：
- `idx_logs_level_time` on `(level, created_at)` — 按等级和时间范围查询
- `idx_logs_module_time` on `(module, created_at)` — 按模块查询

**日志等级对照表**：

| 等级值 | 名称 | 含义 | 典型场景 |
|--------|------|------|----------|
| 0 | `常规` (INFO) | 系统正常运行的信息 | 用户登录、充电开始/结束、账单生成、服务启动/关闭 |
| 1 | `注意` (NOTICE) | 需要关注但不影响运行 | 调度策略切换、修改充电参数、管理员操作记录 |
| 2 | `警告` (WARN) | 潜在问题，系统可自动恢复 | 排队区容量不足、充电协议切换失败（重试后可恢复） |
| 3 | `严重` (ERROR) | 功能受损，需人工介入 | 充电桩故障、账单计算异常、数据库写入失败 |
| 4 | `崩溃` (CRITICAL) | 系统无法继续运行 | 系统启动失败、关键服务异常终止、数据库连接丢失 |

**模块对照表**：

| 模块名 | 对应服务 | 日志覆盖范围 |
|--------|----------|-------------|
| `SYSTEM` | 系统启动/关闭/配置 | 服务启停、配置加载、数据库初始化 |
| `ACCOUNT` | AccountService | 用户注册、登录、密码修改、账号状态变更 |
| `DISPATCH` | DispatchService | 充电桩分配计算、策略执行、匈牙利算法过程 |
| `QUEUE` | QueueService | 车辆入队/出队/换队、三区流转、排队位置变化 |
| `BILLING` | BillingService | 费用计算、账单生成、分段详单拆分 |
| `MONITOR` | MonitorService | 充电桩状态采集、定时刷新、累计统计更新 |
| `ADMIN` | 管理员操作 | powerOn/Off、setParameters、移动车辆、启停充电桩 |
| `FAULT` | 故障处理 | 故障检测、五种策略的重新调度过程、恢复结果 |
| `SCHEDULER` | ScheduledTaskService | 定时任务执行、超时检查、轮询触发 |
| `API` | REST API 层 | 请求入参记录、响应状态、异常捕获 |

## 四、数据库创建 SQL（SQLite）

以下是完整的数据库建表 SQL 脚本（SQLite 方言），可直接用于初始化数据库。

```sql
-- ============================================
-- 智能充电桩调度计费系统 — 数据库初始化脚本
-- 数据库引擎：SQLite
-- ============================================

-- 开启外键约束
PRAGMA foreign_keys = ON;

-- ----------------------------
-- 1. 用户账号表
-- ----------------------------
CREATE TABLE IF NOT EXISTS users (
    id                  INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id             VARCHAR(32)     NOT NULL UNIQUE,
    user_name           VARCHAR(50)     NOT NULL,
    license_plate       VARCHAR(20)     NOT NULL UNIQUE,
    password_hash       VARCHAR(128)    NOT NULL,
    membership_level    INTEGER         NOT NULL DEFAULT 0,
    account_status      VARCHAR(16)     NOT NULL DEFAULT 'INACTIVE',
    created_at          DATETIME        NOT NULL,
    updated_at          DATETIME        NOT NULL,
    last_login_at       DATETIME
);
CREATE INDEX idx_users_license_plate ON users(license_plate);

-- ----------------------------
-- 2. 车辆信息表
-- ----------------------------
CREATE TABLE IF NOT EXISTS vehicles (
    id                      INTEGER PRIMARY KEY AUTOINCREMENT,
    vehicle_id              VARCHAR(32)     NOT NULL UNIQUE,
    user_id                 VARCHAR(32)     NOT NULL REFERENCES users(user_id),
    license_plate           VARCHAR(20)     NOT NULL UNIQUE,
    battery_capacity_kwh    DECIMAL(8,2)    NOT NULL,
    current_battery_percentage DECIMAL(5,2) NOT NULL DEFAULT 0,
    charging_protocol       VARCHAR(20)     NOT NULL,
    created_at              DATETIME        NOT NULL,
    updated_at              DATETIME        NOT NULL,
    CHECK (current_battery_percentage BETWEEN 0 AND 100)
);
CREATE INDEX idx_vehicles_user_id ON vehicles(user_id);

-- ----------------------------
-- 3. 充电桩表
-- ----------------------------
CREATE TABLE IF NOT EXISTS charging_piles (
    id                      INTEGER PRIMARY KEY AUTOINCREMENT,
    pile_id                 VARCHAR(16)     NOT NULL UNIQUE,
    pile_name               VARCHAR(50)     NOT NULL,
    type                    VARCHAR(16)     NOT NULL,
    max_power_kw            DECIMAL(6,2)    NOT NULL,
    status                  VARCHAR(16)     NOT NULL DEFAULT 'AVAILABLE',
    total_charge_num        INTEGER         NOT NULL DEFAULT 0,
    total_charge_time       DECIMAL(10,2)   NOT NULL DEFAULT 0,
    total_capacity          DECIMAL(12,2)   NOT NULL DEFAULT 0,
    parking_spots           INTEGER         NOT NULL DEFAULT 1,
    current_charging_count  INTEGER         NOT NULL DEFAULT 0,
    created_at              DATETIME        NOT NULL,
    updated_at              DATETIME        NOT NULL
);
CREATE INDEX idx_piles_status ON charging_piles(status);

-- ----------------------------
-- 4. 充电桩协议关联表
-- ----------------------------
CREATE TABLE IF NOT EXISTS pile_protocols (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    pile_id     VARCHAR(16)     NOT NULL REFERENCES charging_piles(pile_id),
    protocol    VARCHAR(20)     NOT NULL,
    created_at  DATETIME        NOT NULL,
    UNIQUE(pile_id, protocol)
);

-- ----------------------------
-- 5. 充电桩计费规则表
-- ----------------------------
CREATE TABLE IF NOT EXISTS pile_tariff_configs (
    id                  INTEGER PRIMARY KEY AUTOINCREMENT,
    pile_id             VARCHAR(16)     NOT NULL UNIQUE REFERENCES charging_piles(pile_id),
    peak_price          DECIMAL(6,4)    NOT NULL DEFAULT 1.2,
    normal_price        DECIMAL(6,4)    NOT NULL DEFAULT 0.8,
    valley_price        DECIMAL(6,4)    NOT NULL DEFAULT 0.4,
    base_service_fee    DECIMAL(6,2)    NOT NULL DEFAULT 5.0,
    time_coefficient    DECIMAL(6,4)    NOT NULL DEFAULT 0.5,
    overtime_penalty    DECIMAL(6,4)    NOT NULL DEFAULT 0.8,
    timeout_base_fee    DECIMAL(6,2)    NOT NULL DEFAULT 5.0,
    created_at          DATETIME        NOT NULL,
    updated_at          DATETIME        NOT NULL
);

-- ----------------------------
-- 6. 充电请求表
-- ----------------------------
CREATE TABLE IF NOT EXISTS charging_requests (
    id                  INTEGER PRIMARY KEY AUTOINCREMENT,
    request_id          VARCHAR(32)     NOT NULL UNIQUE,
    car_id              VARCHAR(20)     NOT NULL,
    pile_id             VARCHAR(16)     NOT NULL REFERENCES charging_piles(pile_id),
    request_time        DATETIME        NOT NULL,
    charging_mode       VARCHAR(16)     NOT NULL,
    target_power_kwh    DECIMAL(8,2)    NOT NULL,
    request_status      VARCHAR(16)     NOT NULL DEFAULT 'QUEUED',
    zone_type           VARCHAR(16)     NOT NULL DEFAULT 'QUEUE_AREA',
    queue_position      INTEGER,
    original_pile_id    VARCHAR(16),
    is_active           BOOLEAN         NOT NULL DEFAULT TRUE,
    created_at          DATETIME        NOT NULL,
    updated_at          DATETIME        NOT NULL
);
CREATE INDEX idx_requests_car_id ON charging_requests(car_id);
CREATE INDEX idx_requests_pile_id_status ON charging_requests(pile_id, request_status);
CREATE INDEX idx_requests_car_id_status ON charging_requests(car_id, request_status);

-- ----------------------------
-- 7. 充电会话表
-- ----------------------------
CREATE TABLE IF NOT EXISTS charging_sessions (
    id                      INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id              VARCHAR(32)     NOT NULL UNIQUE,
    request_id              VARCHAR(32)     NOT NULL REFERENCES charging_requests(request_id),
    car_id                  VARCHAR(20)     NOT NULL,
    pile_id                 VARCHAR(16)     NOT NULL,
    start_time              DATETIME        NOT NULL,
    end_time                DATETIME,
    target_power_kwh        DECIMAL(8,2)    NOT NULL,
    charged_power_kwh       DECIMAL(8,2)    NOT NULL DEFAULT 0,
    current_power_kw        DECIMAL(6,2),
    charging_protocol       VARCHAR(20)     NOT NULL,
    fault_interrupted       BOOLEAN         NOT NULL DEFAULT FALSE,
    interrupted_power_kwh   DECIMAL(8,2),
    session_status          VARCHAR(16)     NOT NULL DEFAULT 'ACTIVE',
    created_at              DATETIME        NOT NULL,
    updated_at              DATETIME        NOT NULL
);
CREATE INDEX idx_sessions_request_id ON charging_sessions(request_id);
CREATE INDEX idx_sessions_car_id ON charging_sessions(car_id);
CREATE INDEX idx_sessions_pile_id ON charging_sessions(pile_id);

-- ----------------------------
-- 8. 账单记录表
-- ----------------------------
CREATE TABLE IF NOT EXISTS billing_records (

    charge_amount       DECIMAL(8,2)    NOT NULL,
    charge_duration     DECIMAL(8,2)    NOT NULL,
    start_time          DATETIME        NOT NULL,
    end_time            DATETIME        NOT NULL,
    total_charge_fee    DECIMAL(10,2)   NOT NULL,
    total_service_fee   DECIMAL(10,2)   NOT NULL,

    period_charge_kwh   DECIMAL(8,2)    NOT NULL,
    unit_price          DECIMAL(6,4)    NOT NULL,
    charge_fee          DECIMAL(10,2)   NOT NULL,
    service_fee         DECIMAL(10,2)   NOT NULL,
    subtotal_fee        DECIMAL(10,2)   NOT NULL,
    created_at          DATETIME        NOT NULL

CREATE INDEX idx_detail_bill_id ON detailed_bills(bill_id);

-- ----------------------------
-- 10. 支付订单表
-- ----------------------------
CREATE TABLE IF NOT EXISTS payment_orders (

    car_id              VARCHAR(20)     NOT NULL,
    total_amount        DECIMAL(10,2)   NOT NULL,
    payment_status      VARCHAR(16)     NOT NULL DEFAULT 'UNPAID',
    payment_method      VARCHAR(16),
    payment_time        DATETIME,

-- ----------------------------
-- 11. 故障事件表

## 六、数据流示例（一次完整充电）

```
1. 用户注册/登录
   → users 表插入/查询

2. 提交充电请求 E_chargingRequest
   → charging_requests 插入（request_status='QUEUED', zone_type='QUEUE_AREA'）
   → 根据配置文件 dispatch.default_algorithm 计算最佳充电桩

3. 自动进入等待区
   → charging_requests 更新 zone_type='WAITING_AREA', request_status='WAITING'

4. 确认开始充电 Start_Charging
   → charging_sessions 插入（session_status='ACTIVE'）
   → charging_requests 更新 zone_type='CHARGING_AREA', request_status='CHARGING'
   → charging_piles 更新 status='CHARGING', current_charging_count+=1

5. 充电中修改电量
   → charging_sessions 更新 target_power_kwh（校验 ≥ charged_power_kwh）

6. 结束充电 End_Charging
   → charging_sessions 更新 end_time, charged_power_kwh, session_status='COMPLETED'
   → charging_piles 更新 status, total_charge_num+=1, total_charge_time+=t, total_capacity+=kwh, current_charging_count-=1
   → charging_requests 更新 request_status='COMPLETED', is_active=FALSE

7. 生成账单
   → billing_records 插入
   → detailed_bills 批量插入（按时段拆分）

8. 支付完成
   → payment_orders 插入/更新
   → billing_records 更新 payment_status='PAID'
```

## 七、Logger 模块设计（编码参考）

### 7.1 设计目标

提供一个统一的日志记录模块，满足：
1. **全系统统一入口**：所有模块通过同一个 Logger 实例记录日志
2. **自动持久化**：每次调用 Logger 自动向 `system_logs` 表写入一条记录
3. **终端输出**：自动判断日志等级是否 >= `config.logging.console_threshold`，是则同步输出到终端
4. **使用简单**：一行代码即可记录日志，无需调用者关心数据库写入和等级判断

### 7.2 接口设计

```python
# Logger 模块伪代码（coding 时参考）

class Logger:
    # 等级常量
    LEVEL_INFO = 0       # 常规
    LEVEL_NOTICE = 1     # 注意
    LEVEL_WARN = 2       # 警告
    LEVEL_ERROR = 3      # 严重
    LEVEL_CRITICAL = 4   # 崩溃

    # 模块常量
    MODULE_SYSTEM = "SYSTEM"
    MODULE_ACCOUNT = "ACCOUNT"
    MODULE_DISPATCH = "DISPATCH"
    MODULE_QUEUE = "QUEUE"
    MODULE_BILLING = "BILLING"
    MODULE_MONITOR = "MONITOR"
    MODULE_ADMIN = "ADMIN"
    MODULE_FAULT = "FAULT"
    MODULE_SCHEDULER = "SCHEDULER"
    MODULE_API = "API"

    def __init__(self, db_session, config):
        # 从配置读取 console_threshold
        self.console_threshold = config.logging.console_threshold

    def log(self, module: str, level: int, detail: str):
        # 1. 写入数据库 system_logs 表（始终执行）
        db_write(module, level, detail)

        # 2. 若 level >= console_threshold，同时输出到终端
        if level >= self.console_threshold:
            print(f"[{level}] [{module}] {detail}")

    # 便捷方法
    def info(self, module, detail):
        self.log(module, 0, detail)

    def notice(self, module, detail):
        self.log(module, 1, detail)

    def warn(self, module, detail):
        self.log(module, 2, detail)

    def error(self, module, detail):
        self.log(module, 3, detail)

    def critical(self, module, detail):
        self.log(module, 4, detail)


# 使用示例
logger = Logger(db, config)

# 常规记录（等级0，低于阈值1，仅入DB）
logger.info("SYSTEM", "[启动] 系统已成功初始化 (piles: 5)")

# 注意记录（等级1 >= 阈值1，入DB+终端）
logger.notice("ADMIN", "[切换策略] 管理员切换分配策略至 BATCH_SHORTEST_TIME")

# 警告记录（等级2 >= 阈值1，入DB+终端）
logger.warn("QUEUE", "[容量告警] 桩 P001 排队区容量已达 90% (45/50)")

# 严重记录（等级3 >= 阈值1，入DB+终端）
logger.error("FAULT", "[故障] 桩 P002 检测到通信故障 (strategy: TIME_ORDER)")

# 崩溃记录（等级4 >= 阈值1，入DB+终端）
logger.critical("SYSTEM", "[崩溃] 数据库连接丢失，系统无法继续运行")
```
