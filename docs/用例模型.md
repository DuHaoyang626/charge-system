| 用例编号 | 用例名称 | 描述 | 主要参与者 |

| UC-04 | 更换充电桩队列 | 用户在排队区自由更换到其他充电桩队列，更换后排至目标队列队尾 | 用户 |
| UC-05 | 修改充电请求 | 修改充电电量或充电模式，系统相应更新队列状态 | 用户 |
| UC-06 | 取消充电请求 | 取消充电请求，系统从队列中移除并释放资源 | 用户 |
| UC-07 | 自动进入等待区 | 排队区排到最前时，车辆自动进入等待区（系统处理，无需用户确认） | 系统 |

| UC-10 | 查看充电进度 | 查看实时充电进度、已充电量、剩余时间等详单信息 | 用户 |
| UC-11 | 结束充电 | 到达指定电量后自动结束充电，或用户主动终止 | 用户 |
| UC-12 | 查看账单 | 查看充电费用概览信息（11个字段） | 用户 |

| UC-13 | 支付结算 | 完成支付，生成支付订单 | 用户 |

#### 1.3.2 管理员主要用例

| 用例编号 | 用例名称 | 描述 | 主要参与者 |
|---------|---------|------|-----------|
| UC-14 | 监控充电站状态 | 定时刷新所有充电桩的工作状态、累计充电次数、累计充电时长、累计充电容量 | 管理员 |
| UC-15 | 查看队列详情 | 查看各充电桩队列中车辆详情（车牌号、电池容量、请求电量、等待时间） | 管理员 |
| UC-16 | 移动车辆位置 | 将车辆移动到任何未满队列的任何区域的任意位置 | 管理员 |
| UC-17 | 控制充电桩可用性 | 设置充电桩状态：常规关闭、故障关闭，以及启停充电桩电源 | 管理员 |
| UC-18 | 修改充电桩参数 | 修改充电桩功率上限、支持协议列表、计费规则及三个时段电价数据 | 管理员 |
| UC-19 | 处理充电桩故障 | 充电桩故障时，按所选策略（优先级调度/时间顺序调度/充电中故障恢复/单次调度最短时长/批量调度最短时长）重新调度受影响车辆 | 管理员 |
| UC-20 | 生成运营报表 | 生成充电量、收入、利用率等运营报表 | 管理员 |
| UC-21 | 导出数据 | 导出报表数据为Excel/PDF格式 | 管理员 |
| UC-23 | 运行充电桩 | 启动充电桩进入运行状态，使其可接受充电请求 | 管理员 |
| UC-41 | 管理调度策略 | 系统启动时根据参数选择默认调度策略，管理员运行期间可动态切换正常分配策略和故障处理策略 | 管理员 |
| UC-42 | 设置启动默认策略 | 系统启动前通过配置文件或启动参数设置初始调度策略 | 管理员 |
| UC-43 | 运行时切换策略 | 管理员在系统运行期间动态切换当前调度策略，切换后后续的调度操作使用新策略 | 管理员 |

### 2.1 注册（UC-22）

```mermaid
sequenceDiagram
    actor User as 用户
    participant Client as 客户端
    participant System as 系统
    participant Account as 账号管理

        Client-->>User: 提示设置密码

        Client->>System: set_pwd(******)
        
        System->>Account: 验证密码强度并存储
        Account-->>System: 密码设置成功
        System-->>Client: Return(1) 注册完成
        Client-->>User: 显示"注册成功，请登录"
```

### 2.3 充电申请（UC-02、UC-03、UC-05、UC-08、UC-10、UC-11）

本用例覆盖作业要求的7条指令。以下按阶段分为三个子图。

#### 2.3.1 提交充电申请与查看队列状态

```mermaid
sequenceDiagram
    actor User as 用户
    participant Client as 客户端
    participant System as 系统
    participant Dispatch as 调度服务
    participant Queue as 队列管理
    
    User->>Client: 选择充电模式、输入目标电量
    Client->>System: E_chargingRequest(car_Id, Request_Amount, Request_Mode)
    
    System->>Dispatch: calculateBestQueue(car_Id, Request_Amount, Request_Mode)
    Dispatch->>Dispatch: 计算所有兼容充电桩的预计完成时间
    Dispatch-->>System: 返回最佳充电桩ID及排队位置
    
    System->>Queue: enqueue(car_Id, pileId, Request_Mode)
    Queue-->>System: 返回排队信息
    System->>System: 记录请求时间request_time
    
    System-->>Client: Return(car_position, car_state="排队中", queue_Num, request_time)
    Client-->>User: 显示排队信息
    
    Note over User,System: --- 修改电量和模式 ---
    
    User->>Client: 修改充电电量
    Client->>System: Modify_Amount(car_Id, Amount)
    System->>Queue: updateTargetAmount(car_Id, Amount)
    alt 修改成功
        System-->>Client: Return(1)
    else 修改失败（电量无效）
        System-->>Client: Return(0)
    end
    
    User->>Client: 修改充电模式
    Client->>System: Modify_Mode(car_Id, Mode)
    System->>Dispatch: recalculateQueue(car_Id, Mode)
    alt 修改成功
        System-->>Client: Return(1)
    else 修改失败（模式不支持）
        System-->>Client: Return(0)
    end
    
    Note over User,System: --- 查看队列状态 ---
    
    User->>Client: 查看当前排队状态
    Client->>System: Query_Car_State(car_id)
    System->>Queue: getCarState(car_id)
    Queue-->>System: 返回排队信息
    System-->>Client: Return(car_Number_before_position, car_state, queue_Num, request_time)
    Client-->>User: 显示排队详情
```

#### 2.3.2 开始充电

```mermaid
sequenceDiagram
    actor User as 用户
    participant Client as 客户端
    participant System as 系统
    participant Pile as 充电桩
    participant Session as 充电会话

        Pile-->>System: 锁定成功
        
        System->>System: 更新请求状态为"充电中"
        System-->>Client: Return(1) 开始充电成功
```

### 2.6 运行充电桩（UC-23、UC-17、UC-18）

本用例覆盖作业要求的4条指令，其中 `powerOn`/`powerOff` 合入 `setPileAvailability`，`setParameters` 合入 `updatePileParameters`。

```mermaid
sequenceDiagram
    actor Admin as 管理员
    participant AdminClient as 管理客户端
    participant System as 系统
    participant Pile as 充电桩
    
    Note over Admin,Pile: --- 启动充电桩 ---
    
    Admin->>AdminClient: 选择充电桩，执行开机
    AdminClient->>System: powerOn(pile_Id)
    
    System->>Pile: 发送开机指令
    Pile-->>System: 返回开机状态
    
    alt 开机失败
        System-->>AdminClient: Return(0) 开机失败
    else 开机成功
        System->>System: 更新充电桩状态为"可用"
        System-->>AdminClient: Return(1) 开机成功
        AdminClient-->>Admin: 显示开机成功
    end
    
    Note over Admin,Pile: --- 设置参数（计费规则+电价数据）---
    
    Admin->>AdminClient: 输入计费规则和三个时段电价
    AdminClient->>System: setParameters(pile_Id, 计费规则, 峰时电价, 平时电价, 谷时电价)
    
    System->>Pile: 更新计费规则配置
    System->>System: 更新分时电价策略
    Pile-->>System: 配置更新成功
    
    System-->>AdminClient: Return(1) 参数设置成功
    AdminClient-->>Admin: 显示参数已更新
    
    Note over Admin,Pile: --- 运行充电桩 ---
    
    Admin->>AdminClient: 启动充电桩进入运行状态
    AdminClient->>System: Start_ChargingPile(pile_Id)
    
    System->>Pile: 检查充电桩就绪状态
    Pile-->>System: 返回就绪状态
    
    alt 充电桩未就绪
        System-->>AdminClient: Return(0) 充电桩未就绪
    else 启动成功
        System->>System: 更新充电桩状态为"运行中"
        System-->>AdminClient: Return(1) 充电桩已进入运行状态
        AdminClient-->>Admin: 显示运行中
    end
    
    Note over Admin,Pile: --- 关闭充电桩 ---
    
    Admin->>AdminClient: 选择充电桩，执行关机
    AdminClient->>System: powerOff(pile_Id)
    
    System->>Pile: 检查充电桩是否正在充电
    Pile-->>System: 返回当前状态
    
    alt 有车辆正在充电
        System-->>AdminClient: Return(0) 充电桩正在使用中，无法关机
    else 关机成功
        System->>Pile: 发送关机指令
        System->>System: 更新充电桩状态为"关闭"
        System-->>AdminClient: Return(1) 关机成功
        AdminClient-->>Admin: 显示已关机
    end
```

### 2.7 查看充电桩状态（UC-14）

要求定时刷新所有充电桩状态并在客户端显示。

```mermaid
sequenceDiagram
    actor Admin as 管理员
    participant AdminClient as 管理客户端
    participant System as 系统
    participant Monitor as 状态监控服务
    participant Pile as 充电桩
    
    Note over Admin,Pile: 管理员主动查询
    
    Admin->>AdminClient: 选择充电桩查看状态
    AdminClient->>System: Query_PileState(pile_Id)
    
    System->>Monitor: getPileStats(pile_Id)
    Monitor->>Pile: 读取实时工作状态
    Pile-->>Monitor: 返回状态数据
    Monitor-->>System: 返回统计数据
    
    System-->>AdminClient: Return(workingState, TotalChargeNum,<br/>TotalChargeTime, TotalCapacity)
    AdminClient-->>Admin: 显示充电桩状态详情
    
    Note over Admin,Pile: 定时刷新机制
    
    loop 每N秒定时刷新
        Monitor->>Pile: 批量采集所有充电桩状态
        Pile-->>Monitor: 返回状态数据
        Monitor->>Monitor: 更新状态缓存
        Monitor-->>System: 推送状态变更
        System-->>AdminClient: 推送刷新数据
        AdminClient-->>Admin: 自动更新显示
    end
```

```mermaid
sequenceDiagram
```

### 3.4 单次调度最短时长 — 故障场景（SSD-F4）

当故障处理策略设为"单次调度最短时长"时，对每台受影响车辆独立计算最优充电桩，贪心选择总完成时间最短的。

```mermaid
sequenceDiagram
    participant Monitor as 监控系统
    participant System as 系统

    System->>Dispatch: getActiveFaultStrategy()

    System->>FaultPile: getAffectedVehicles(pile_Id)
    FaultPile-->>System: 返回所有受影响车辆列表 [V₁, V₂, ..., Vₙ]

        Dispatch->>Dispatch: getCompatiblePiles(Vᵢ.protocol, Vᵢ.mode)
        Dispatch-->>Dispatch: 返回同类型可用桩列表 [P₁, P₂, ..., Pₙ]
        
        loop 遍历每个可用充电桩 Pⱼ
            Dispatch->>Dispatch: 计算 wⱼ(Pⱼ当前排队总充电时间)
            Dispatch->>Dispatch: 计算 cⱼ(Vᵢ.Request_Amount / Pⱼ.maxPowerKW)
            Dispatch->>Dispatch: 计算 Tⱼ = wⱼ + cⱼ
        end
        
        Dispatch->>Dispatch: 选择 min(Tⱼ) 对应的充电桩 P_best
        
        Dispatch->>TargetPile: assignVehicle(Vᵢ, P_best)
        TargetPile-->>Dispatch: 分配成功
    end
    
    Dispatch-->>System: 返回调度结果列表
    System->>System: 更新故障充电桩状态为"故障关闭"
    System->>System: 发送调度结果通知给管理员
```

#### 3.4.1 策略对比：与传统 SCTF 的区别

| 维度 | 传统 SCTF（最短充电时间优先） | 本策略（单次调度最短时长） |
|------|---------------------------|-------------------------|
| 考虑因素 | 仅考虑充电时长 | 排队等待时长 + 充电时长 |
| 调度触发 | 充电桩空闲时选择最短充电车辆 | 车辆到达时选择最优充电桩 |
| 公平性 | 长充电车辆可能被饿死 | 总时长最短兼顾等待，不易饿死 |

#### 3.4.2 计算过程示例

以下展示车辆A（60kWh，快充模式）提交请求时，两个候选充电桩的完成时间计算：


#### 3.4.3 边界情况处理

**情况1：有完全空闲的同类型充电桩**

当存在空闲充电桩时，等待时间 w₀ = 0，总时间 T₀ = c₀（直接充电时间），该值一定小于任何有排队等待的充电桩的总时间，因此直接选择空闲桩。

**情况2：所有兼容充电桩总时长相等**

当 T₁ = T₂ = ... = Tₙ 全部相等时，使用二次规则：选择排队车辆最少（或编号最小）的充电桩。该规则确保在最优解相同时有确定的仲裁逻辑，避免系统随机选择。

（注：此策略作为**正常分配算法**时默认启用，实现于 `assignByShortestTotalTime` 契约中）

### 3.5 批量调度最短时长 — 故障场景（SSD-F5）

当故障处理策略设为"批量调度最短时长"时，将所有受影响车辆视为一个批量，用匈牙利算法求全局最优分配。

```mermaid
sequenceDiagram
    participant Monitor as 监控系统
    participant System as 系统
    participant Dispatch as 调度服务
    participant FaultPile as 故障充电桩
    participant TargetPile as 目标充电桩
    
    Monitor->>System: detectFault(pile_Id)
    System->>System: 记录故障事件
    
    Note over System,Dispatch: 获取当前激活的故障处理策略
    System->>Dispatch: getActiveFaultStrategy()
    Dispatch-->>System: 当前策略 = "批量调度最短时长"
    
    System->>FaultPile: getAffectedVehicles(pile_Id)
    FaultPile-->>System: 返回所有受影响车辆列表 [V₁, V₂, ..., Vₙ]
    
    System->>Dispatch: getAvailableCompatiblePilesForBatch(vehicles)
    Dispatch-->>System: 返回可用充电桩列表 [P₁, P₂, ..., Pₘ]
    
    System->>Dispatch: batchAssignByShortestTotalTime(vehicles, piles)
    
    Note over Dispatch: 步骤1：构建 N×M 成本矩阵
    Dispatch->>Dispatch: 创建矩阵 C[N][M]
    
    loop 车辆 Vᵢ (i=1 to N)
        loop 充电桩 Pⱼ (j=1 to M)
            Dispatch->>Dispatch: 计算 C[i][j] = wⱼ + (Vᵢ.Request_Amount / Pⱼ.maxPowerKW)
        end
    end
    
    Note over Dispatch: 步骤2：执行匈牙利算法求全局最优
    Dispatch->>Dispatch: hungarianAlgorithm(C[N][M])
    Note right of Dispatch: 算法过程：<br/>1. 行归约 → 2. 列归约<br/>3. 最小直线覆盖零<br/>4. 若直线数 < N，调整后重复<br/>5. 取 N 个独立零
    
    Dispatch-->>System: 返回最优分配方案<br/>[(V₁→P₃), (V₂→P₁), ..., (Vₙ→Pₖ)]
    
    Note over System,TargetPile: 步骤3：批量执行分配
    
    par 并行分配每台车辆
        loop 每个车辆→充电桩对
            System->>TargetPile: assignVehicle(Vᵢ, Pⱼ)
            TargetPile-->>System: 分配成功
        end
    end
    
    System->>System: 更新故障充电桩状态为"故障关闭"
    System->>System: 记录批量调度结果（总完成时间 = ΣTᵢⱼ）
    System->>System: 发送批量调度结果通知给管理员
```

#### 3.5.1 贪心算法 vs 全局最优算法对比

以下展示批量调度中贪心算法（等同于逐一执行单次最短）与匈牙利算法（全局最优）的差异。

**场景：** 3台车辆 V₁(80kWh), V₂(30kWh), V₃(20kWh)；3个充电桩 P₁(60kW), P₂(120kW), P₃(120kW)；P₁排队已有1辆车(30kWh)，P₂/P₃空闲

**成本矩阵 C[3×3]：**

| 车辆\充电桩 | P₁ (60kW，有排队) | P₂ (120kW，空闲) | P₃ (120kW，空闲) |
|------------|-----------------|-----------------|-----------------|
| V₁ (80kWh) | (30/60+80/60)=1.83h | (0+80/120)=0.67h | (0+80/120)=0.67h |
| V₂ (30kWh) | (30/60+30/60)=1.00h | (0+30/120)=0.25h | (0+30/120)=0.25h |
| V₃ (20kWh) | (30/60+20/60)=0.83h | (0+20/120)=0.17h | (0+20/120)=0.17h |

- **贪心结果**（逐一分配）：V₁→P₂(0.67h), V₂→P₃(0.25h), V₃→P₁(0.83h) → 总时间 = 1.75h
- **全局最优**（匈牙利算法）：V₁→P₃(0.67h), V₂→P₂(0.25h), V₃→P₁(0.83h) → 总时间 = 1.75h

> 本算例中两者结果一致（因空闲桩规格相同），但在非对称场景中匈牙利算法通常能节省约 10~20% 的总完成时间。

#### 3.5.2 边界情况处理

**情况1：待调度车辆数 > 可用充电桩数（N > M）**

当 N=10, M=5 时，分批处理：从 10 辆中选出 5 辆使成本矩阵总和最小的组合进行最优分配，剩余 5 辆加入全局等待队列队首（优先于新请求）。匈牙利算法每次处理一个 N×M 子矩阵，直至全部车辆分配完毕或再无可用充电桩。

**情况2：车辆需求与充电桩不兼容**

当车辆 Vᵢ 的充电协议不被充电桩 Pⱼ 支持时，在成本矩阵中将 C[i][j] 设为 ∞，匈牙利算法自动跳过此分配组合。

  2. 系统运行中
  3. strategyType 必须是有效的分配策略标识符
- **后置条件**：
  1. 系统当前激活的分配策略切换为 strategyType
  2. 后续所有正常分配操作使用新策略
  3. 记录策略切换日志
- **异常处理**：
  - 策略标识符无效：Return(0)，策略不变

#### 契约26：switchFaultStrategy(faultType)

- **操作**：switchFaultStrategy(faultType: String)
- **交叉引用**：UC-43 运行时切换策略
- **前置条件**：
  1. 管理员已登录
  2. 系统运行中
  3. faultType 必须是有效的故障策略标识符
- **后置条件**：
  1. 系统当前激活的故障处理策略切换为 faultType
  2. 后续所有故障处理操作使用新策略
  3. 记录策略切换日志
- **异常处理**：
  - 策略标识符无效：Return(0)，策略不变

## 六、总结

本用例模型按第二次作业要求进行了全面更新，主要变更包括：

1. **新增用例**：注册（UC-22）、查看详单（UC-12a）、运行充电桩（UC-23）、管理调度策略（UC-41）
2. **登录分离**：登录（UC-01）与注册（UC-22）独立为两个用例，登录使用 car_Id + password
3. **SSD 全面对齐**：7个用例 + 1个子用例 + 1个策略管理的系统顺序图，消息名和参数严格按照作业指令表命名
4. **五种故障调度交互图**：优先级调度（SSD-F1）、时间顺序调度（SSD-F2）、充电中故障恢复（SSD-F3）、单次调度最短时长（SSD-F4）、批量调度最短时长（SSD-F5），各自独立
5. **操作契约扩展**：27个操作契约覆盖全部指令（含故障处理、策略管理和正常分配算法），含前置条件、后置条件、异常处理
6. **调度策略可切换**：系统通过启动参数决定默认策略，管理员运行期间可动态切换正常分配策略和故障处理策略
7. **指令对应表**：完整的用例-指令-参数-返回格式表，含策略管理指令

该模型为系统动态结构设计提供了完整、可编码的指令级规范。
